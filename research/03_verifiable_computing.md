# AI推理计费透明化：可验证计算与审计技术深度调研报告

## 核心发现

**当前没有任何单一技术能够在生产环境中同时实现"实时交互延迟"和"密码学级可验证性"**。最可行的方案是分层架构：客户端token计数校验 + TEE硬件证明 + 数字签名审计日志 + 周期性ZK审计。

---

## 1. 零知识证明（ZKP）

### 1.1 zk-SNARKs vs zk-STARKs

| 维度 | zk-SNARKs | zk-STARKs |
|------|-----------|-----------|
| 证明生成 | 55 ms | 3,809 ms |
| 验证时间 | 1,807 ms | 472 ms |
| 证明大小 | 384 bytes | ~69 KB |
| 可信设置 | 需要 | 无需 |
| 后量子安全 | 否 | 是 |

### 1.2 ZKP在AI推理中的现状

| 项目 | 描述 | 状态 |
|------|------|------|
| **Lagrange DeepProve-1** | 首个生产级全LLM推理ZK系统 | 比EZKL基线快158倍 |
| **ZIP** | 高精度ZK-SNARK，原生浮点 | 非线性激活电路减少3个数量级 |
| **zkPyTorch** | PyTorch到ZK直接编译器 | VGG-16仅需2.2秒 |
| **EZKL** | ONNX到zk-SNARK | MNIST级~2秒，Trail of Bits审计通过 |

**结论**：ZKP实时验证**目前不可行**（证明生成分钟级），但适合离线审计。

## 2. 可信执行环境（TEE）

### 2.1 主要TEE技术

| 技术 | 隔离粒度 | GPU支持 | 2025成熟度 |
|------|---------|---------|-----------|
| Intel TDX | VM级 | 是（H100+） | 生产级 |
| AMD SEV-SNP | VM级 | 是（H100/H200） | 生产级 |
| NVIDIA CC | GPU级 | 原生 | 生产级 |
| AWS Nitro Enclaves | 进程级 | 否 | 生产级 |

### 2.2 性能开销

**Microsoft Azure Confidential AI（2025）**：
- NCCads_H100_v5系列：AMD SEV-SNP + NVIDIA H100
- 性能开销：**<5%**（达到原生95-99%性能）
- Azure OpenAI Service支持机密计算

**NVIDIA H100/H200 CC**：
- 小模型：~5-9%开销
- 大模型：~0%开销

### 2.3 局限性

- 侧信道攻击（缓存时序、功耗分析）
- 预计算噪声复用（LLM推理中静态密钥跨查询）
- 物理访问风险

## 3. 远程证明

### 3.1 验证流程

```
1. 客户端发起推理请求
2. 服务端TEE生成证明报告（模型哈希、运行时度量）
3. 证明报告发送给验证服务
4. 验证服务返回签名的证明令牌
5. 客户端验证模型哈希 == 预期值 && TEE状态 == 有效
```

### 3.2 在计费中的应用

- **模型真实性验证**：防止"模型替换"欺诈
- **推理环境验证**：确保无篡改执行
- **计费完整性**：结合TEE内计数器与证明

## 4. 客户端独立计量

### 4.1 Tokenizer开放现状

| 模型 | Tokenizer | 客户端可用性 |
|------|-----------|-------------|
| GPT系列 | tiktoken | ✅ 可用 |
| Llama | SentencePiece | ✅ 可用 |
| Claude | 专有 | ❌ 无官方库 |
| Gemini | SentencePiece | ⚠️ 有限支持 |

### 4.2 实测差异

| 场景 | 客户端计数 | 服务端计数 | 差异 |
|------|-----------|-----------|------|
| OpenAI官方 | ~12K | ~19K | **~58%** |
| LangSmith vs tiktoken | 2,067 | 11,185 | **~5.4x** |

**差异来源**：系统提示、对话历史、函数定义未计入客户端；消息格式化开销；不同tokenizer版本。

### 4.3 推荐阈值

- **硬阈值（5%）**：触发自动告警和人工审核
- **软阈值（10%）**：触发日志记录和趋势分析
- **校准机制**：每周用已知输入交叉验证

## 推荐分层架构

```
Layer 4: 周期性ZK审计（离线）
  - 每月抽样验证
  - 5年内逐步引入实时ZK

Layer 3: 数字签名 + Merkle审计日志
  - 每请求签名计费记录
  - 批量Merkle根上链

Layer 2: TEE + 远程证明
  - GPU机密计算
  - 运行时证明含模型哈希

Layer 1: 客户端独立计量
  - 客户端本地token计数
  - 与服务端对比（阈值±5%）
```

**实施优先级**：
1. **立即**：客户端token计数 + 数字签名日志（零额外成本）
2. **短期（6-12月）**：TEE部署（主流云已支持）
3. **中期（1-2年）**：远程证明集成
4. **长期（3-5年）**：ZK审计作为补充验证层
